# 全域 AI 開發守則

> 本檔是跨專案共用的母版。每個專案應同步一份到 `docs/90_全域AI開發守則.md`，讓各種 AI 工具都能直接讀取 repo 內副本。

## 目的

整理跨專案都適用的開發錯誤經驗，降低重複犯錯，並讓不同 AI 工具在同一個 repo 中讀到一致的守則。

文中出現的特定框架或工具（例如 Playwright、Vitest、Next.js、Python mypy）僅作為**可遷移原則的範例**；請依各專案技術棧決定是否直接套用，不必視為「所有語言／所有專案型態一律同等適用」。

## 核心守則

### 1. 先分類失敗，再決定修改點

遇到問題先判斷它屬於哪一類，再決定要改哪裡：

- 產品邏輯
- 測試資料或 seed
- mock 不完整
- 執行環境或權限
- 背景程序與檔案鎖定
- 測試分層設計

不要一看到錯誤就直接改產品程式。

### 2. 修測試前先看完整體共享資料

單一測試失敗，不代表只影響單一案例。

- 先看完整個測試檔
- 先看共享 fixture、seed、helper
- 盡量做最小修改
- 修改後重跑受影響範圍與整包測試

### 3. mock 必須對齊真實介面行為

如果測試依賴鏈式查詢、分頁、篩選、錯誤格式或回傳結構，mock 就要同步支援。

常見誤判是：

- 產品沒壞，mock 少了一段鏈式能力
- 測試資料沒壞，回傳 shape 和真實實作不一致

### 4. 保持測試邊界清楚

把不同層級的驗證拆開：

- `unit` 驗純邏輯
- `integration` 驗流程與資料整合
- `UI` 驗互動
- `smoke` 驗真實外部系統

需要真實外部系統的測試，不要混入平常的本地快速測試。

### 5. build 或執行失敗先查環境

看到這些線索時，先查程序、權限與輸出路徑：

- `EPERM`
- `Permission denied`
- `spawn`
- 無法清空輸出目錄
- 檔案被使用中

優先檢查：

- 是否仍有 dev server 或背景程序
- 是否有建置相關程序仍佔用輸出檔案（例如 Node、Java、.NET／MSBuild 等編譯或打包程序）
- 是否是沙箱、權限或路徑問題

### 6. 擴功能優先沿用既有資料流

新增功能時，先找目前專案已存在的：

- service
- type
- data source
- workflow

除非真的必要，不要平行發明第二套流程。

### 7. AI 工具入口檔保持薄

`AGENTS.md`、`CLAUDE.md`、`.cursor/rules`、`.agent/rules`、`.github/copilot-instructions.md` 應只放摘要與指向共用文件的入口，不要各自維護大段規則。

### 8. 前端顯示優先使用人類可讀名稱；無對照時可暫示代碼並標示

**規則（強制）**

任何呈現給**目標使用者語系**的介面（表格、下拉選單、標籤、篩選條件），凡涉及系統內部代碼（`snake_case`、`UPPER_CASE`、UUID、英文縮寫等），**預設應**轉換為對應的可讀名稱（繁中產品即為中文標籤）後再顯示。若尚無對照表或後端未提供顯示名稱，**允許暫時顯示原始代碼**，但須有視覺提示（例如灰色字、標註「未對照」），且後續應補上對照或資料來源；不得留空白或顯示無意義的錯誤占位。

**多語系產品**：本條以「使用者當前語系的可讀標籤」為準，不要求一律繁中；核心要求是**不應毫無脈絡地直接丟出內部代碼**。

**常見需要轉換的場景**

| 情境 | 禁止顯示 | 正確顯示 |
|---|---|---|
| 交通方式 | `car_gasoline` | `汽油小客車` |
| 操作類型 | `recalc_completed` | `重算完成` |
| 目標類型 | `recalculation_job` | `重算任務` |
| 使用者角色 | `esg_manager` | `ESG 管理員` |
| 月份狀態 | `closing` | `處理中` |
| 任何 UUID | `dc915c45-ffe3-...` | 對應的名稱或帳號 |

**實作標準**

1. 在頁面頂端定義 `const LABEL_MAP: Record<string, string> = { ... }` 維護代碼對照表。
2. 資料庫中有顯示名稱欄位（如 `mode_name`、`department_name`）時，優先從 API 取得，不要在前端硬編碼。
3. 無對照時的 fallback 須符合本條開頭說明（暫示代碼須有視覺提示、不得空白占位），並應排程補對照或後端欄位。
4. 下拉選單、多選元件必須在 `label` 呈現可讀名稱、`value` 保留原始代碼，確保送給後端的仍是系統代碼。

**反例（禁止）**

```tsx
// ❌ 直接顯示系統代碼
<td>{row.transport_mode}</td>
<td>{item.action}</td>

// ✅ 轉換為中文
<td>{transportMap[row.transport_mode] ?? row.transport_mode}</td>
<td>{ACTION_LABELS[item.action] ?? item.action}</td>
```

### 9. API 路由定義必須「靜態路徑先、動態路徑後」

同一層級若同時存在：

- `/manage/customer-companies`（靜態）
- `/{tenant_id}`（動態）

必須先註冊靜態路由，再註冊動態路由，避免靜態請求被動態參數吃掉，導致 404/422 或打到錯誤 handler。

### 10. 多租戶代管頁的 tenant context 必須端到端一致

當「A 租戶管理員」可代管「B 租戶資料」時：

- 前端每次請求都要明確帶目標 tenant id（例如 `X-Tenant-Id`）
- 後端不可只用登入者 tenant id 直接查資料
- 後端必須驗證跨租戶權限（未授權回 403）

否則會出現「資料本身正確，但代管畫面顯示別的租戶資料」的假象。

### 11. 危險操作語意必須和實際行為一致（停用 vs 永久刪除）

若 API 只做 `is_active=false`，UI 與文案必須明確寫「停用」；
若需求是資料不可復原，才可提供「永久刪除」，且需：

- 明確確認視窗（不可復原）
- 後端實作真正刪除與相依資料處理
- 針對核心租戶或保留資料加保護條件

禁止「按鈕寫刪除，實際只停用」。

### 12. 建立流程遇到資料庫鎖衝突時，必須回可行動的錯誤訊息

前端看到 `Failed to fetch` 常是後端 500 被抽象化後的結果。對建立/異動流程，後端需將常見 DB 例外（如 deadlock、unique constraint）轉成可理解且可重試的業務錯誤（409/422），前端需顯示可讀訊息並避免重複送出。

### 13. 修改含自然語言或多位元組字元後，應掃描並修復亂碼字串

**規則（強制；觸發範圍限縮）**

下列情況**必須**執行亂碼檢查（**不必**每改一行與字串無關的程式就跑）：

- 已修改含有 **中文、日文等 UTF-8 多位元組字元** 的字串字面量、使用者可見文案、錯誤訊息常數的檔案。
- 使用 AI 工具對上述檔案進行**多輪反覆編輯**之後。

AI 工具反覆編輯這類內容時，字串可能損壞成連續 `?????`；靜態分析（lint、型別檢查、建置）通常**無法偵測**，往往在執行時才被發現。

**建議掃描方式（以 Node.js／JavaScript 為例）**

以下腳本將 `TARGET_FILE.js` 換成實際路徑後執行。**其他語言專案**請以相同精神撰寫對等腳本（掃描字串字面量中的異常 `?` 或非法位元組），不必拘泥於 Node。

```bash
node -e "
const fs = require('fs');
const content = fs.readFileSync('TARGET_FILE.js', 'utf8');
const lines = content.split('\n');
let count = 0;
lines.forEach((line, i) => {
  const t = line.trimStart();
  if (t.startsWith('//') || t.startsWith('*')) return;
  if (/['\`][^'\`\n]*\?\?[^\`'\n]*['\`]/.test(line)) {
    console.log((i+1) + ': ' + line.trim().substring(0,100));
    count++;
  }
});
console.log('Garbled strings found:', count);
"
```

**掃描結果僅供輔助**：正規式可能與合法字串（例如說明文字中刻意出現的連續問號）碰撞，**必須人工複核**後再修改，避免機械式誤刪。

**判斷標準**

| 情況 | 判斷 | 處理 |
|------|------|------|
| `'?????'`、`'帳號??錯誤'`（連續兩個以上 `?`） | 亂碼 | **立即修復** |
| URL querystring `?key=value`、三元運算子 `a ? b : c` | 正常 | 忽略 |
| charCode 128–159 混入字串（編輯工具難以字串搜尋匹配）| 位元組損壞 | 用語言原生腳本檢查並替換 |

**修復優先順序**

1. **P0** — 使用者可見字串（`error:` 回傳、介面訊息、`answer:` 等）→ 本次提交前必修
2. **P1** — 開發者 log（`console.error`、`console.warn`、`hint:`）→ 本次工作階段內修復
3. **P2** — 程式碼註解 → 可選修復

## 快速檢查清單

開始除錯前，先問自己：

1. 這次錯誤比較像產品、測試、mock，還是環境？
2. 我有先看完整個測試檔與共享資料嗎？
3. 這個驗證屬於正確的測試層級嗎？
4. 我現在是在延伸既有資料流，還是在偷偷重做一套？
5. 有沒有先排除背景程序、檔案鎖定、權限問題？

### [2026-04-12] AI 工具反覆編輯後中文字串變成問號亂碼
- scope: global
- project: SCS-ServiceAI-JAVA
- area: encoding / string-corruption
- symptom: 使用者登入密碼錯誤時，畫面顯示大量問號而非正常中文錯誤訊息；問答結果末尾出現信心分數顯示異常；多處中文字串一併損壞。
- root_cause: 原始碼經多輪語法修復後 UTF-8 中文字元損壞，部分字串含非標準位元組（例如 0x80），一般字串搜尋難以匹配。
- wrong_assumption: 以為只要語法檢查與 lint 通過，字串內容就一定正確；靜態工具通常無法偵測此類損壞。
- checks_next_time: 每次修改含字串常數的檔案後執行亂碼掃描（見核心守則第 13 條）。若編輯工具回報「找不到字串」，先用腳本檢查 charCode，若出現 128–159 區間位元組，改以該語言腳本替換。
- rule: 凡以 AI 工具修改含 UTF-8 多位元組字元的原始碼後，必須執行亂碼掃描；發現連續異常問號應立即修復，不得拖到執行時才處理。

### [2026-04-06] E2E 修失敗前先確認真實 API 路徑與 payload
- scope: global
- project: AIHR_LAW
- area: testing
- symptom: E2E 顯示上傳流程失敗，看起來像產品壞掉。
- root_cause: 測試仍 mock 舊端點，實際頁面已改走新的 API wrapper 與 request shape。
- wrong_assumption: 直接假設失敗來自產品邏輯，而不是測試端點漂移。
- checks_next_time: 先看當前頁面實際呼叫的 API wrapper、路徑與 payload，再決定要改產品還是改 mock。
- rule: 任何 E2E request 失敗，先確認「當前 UI 真正發出的 request」是否仍和測試 mock 對齊，不要先改產品程式。

### [2026-04-06] Playwright route mock 不可用重疊 pattern 互相覆蓋
- scope: global
- project: AIHR_LAW
- area: mock
- symptom: 同一路徑的測試回應行為不穩定，修改 helper 順序就會改變結果。
- root_cause: 同時存在專用 route 與 catch-all route，且後註冊 route 會覆蓋前者。
- wrong_assumption: 以為多寫幾個相近 mock 只會增加保險，不會改變實際命中順序。
- checks_next_time: 搜同一路徑的所有 route pattern，確認只有一個 authoritative handler，或 pattern 明確互斥。
- rule: Playwright mock 對同一資源只保留一套權威 route；若必須分流，pattern 必須明確不重疊。

### [2026-04-06] 初次載入與背景刷新必須分開處理
- scope: global
- project: AIHR_LAW
- area: ui
- symptom: 頁面輪詢或 websocket refresh 時，整塊資料區反覆跳回 loading，E2E 也變得不穩。
- root_cause: 把 `isFetching` 與 `isLoading` 混成同一個全頁 loading 狀態。
- wrong_assumption: 背景刷新和首載都屬於同一種 loading UX。
- checks_next_time: 檢查 query 是否有 polling 或 background refetch；若有，主內容只應受初次載入控制。
- rule: 有背景刷新時，內容區只看首載 loading；背景 refetch 只能更新按鈕、spinner 或小提示，不能把已渲染內容整塊清掉。

### [2026-04-06] E2E 斷言要綁使用者契約，不綁內部實作與位置
- scope: global
- project: AIHR_LAW
- area: testing
- symptom: 測試因 API path 字串、`first()`、`nth()`、寬鬆 heading 匹配而脆弱或誤判。
- root_cause: 斷言依賴內部實作細節，沒有綁在使用者真正可見的區塊與互動上。
- wrong_assumption: 只要畫面上能找到文字或第一個按鈕，測試就算穩定。
- checks_next_time: 優先用 section heading、row scope、dialog scope、href + click + waitForURL 來限定範圍。
- rule: E2E 應驗證使用者契約與頁面結構，不應依賴內部 API path 文本或位置型 selector。

### [2026-04-06] Dev server 下的長流程 E2E 不要依賴 `waitUntil: "load"`
- scope: global
- project: AIHR_LAW
- area: testing
- symptom: `page.goto()` 在 Next.js dev server 下出現 timeout、frame detached 或偶發全頁 reload。
- root_cause: dev server 熱重載與額外資源會讓 `load` 事件不穩，長流程更容易撞到這種抖動。
- wrong_assumption: `load` 一定比 `domcontentloaded` 更安全。
- checks_next_time: 多頁 E2E 在 dev mode 優先使用 `domcontentloaded`，再加頁面專屬 ready assertion；必要時提高 test timeout。
- rule: 長流程 Playwright 在 dev server 環境應以 `domcontentloaded` + 明確頁面 ready condition 驗證，不要把穩定性綁在 `load` 事件上。

### [2026-04-06] 測試必須證明副作用真的發生，不能只驗本地暫存狀態
- scope: global
- project: AIHR_LAW
- area: testing
- symptom: 上傳測試只看到已選檔名就通過，無法證明 server mutation 後清單真的刷新。
- root_cause: 斷言命中了 mutation 前就存在的 local state，而不是 mutation 造成的 server-driven UI 變化。
- wrong_assumption: 畫面出現檔名就等於上傳成功並刷新列表。
- checks_next_time: mutation 後至少驗其中一項：refetch 次數、新 row、status 變化、成功訊息、後續按鈕狀態。
- rule: 所有 mutation 測試都必須驗證「由 server 回應或 refetch 造成的副作用」，不能只驗證本地輸入狀態。

### [2026-04-06] 最終 sign-off 必須區分本次變更驗證與 repo 既有失敗
- scope: global
- project: AIHR_LAW
- area: workflow
- symptom: 指定範圍的變更與 targeted tests 都通過，但 repo-wide `tsc`/`lint` 仍因既有問題失敗。
- root_cause: 專案早已存在與本次 diff 無關的靜態檢查錯誤。
- wrong_assumption: 只要這次修的測試綠了，就可以宣告整個 repo 靜態簽核完成。
- checks_next_time: 回報時分開寫 targeted verification 與 repo-wide health，明確標示哪些是本次改動、哪些是既有技術債。
- rule: 最終回報要清楚區分「本次變更已驗證通過」與「整個 repo 仍有既有靜態錯誤」，不得混為一談。

### [2026-04-06] Python `# type: ignore` 是最後手段，不是預設解法
- scope: global
- project: AIHR_LAW
- area: typing
- symptom: 型別檢查出現 `arg-type` 或 `assignment` 錯誤，用 `# type: ignore` 快速壓過。
- root_cause: 沒有修正真正的型別宣告，只是讓 mypy 閉嘴；錯誤照樣存在，只是看不到。
- wrong_assumption: 加 `# type: ignore` 和修正型別是等效的兩種選擇。
- checks_next_time:
  - `dict.get()` 回傳 `Optional[V]`，用 `lambda k: d[k]` 而非 `d.get` 傳給需要 `V` 的 `key=`
  - 若值其實是 `tuple[type, ...]`（多型別 isinstance），就宣告 `dict[str, tuple[type, ...]]`，不要宣告 `dict[str, type]`
  - 函式回傳值是有限字面量集合，應宣告 `Literal["A", "B", "C"]`
  - 從 `Any` 窄化為已知型別，用 `cast()` 而非 `# type: ignore`
- rule: 每個 `# type: ignore` 都必須有理由；若根因是型別宣告錯誤，一律修正宣告，不加 ignore 留坑。

### [2026-04-06] `except Exception` 必須帶 log，不能靜默吞掉
- scope: global
- project: AIHR_LAW
- area: error-handling
- symptom: WebSocket 斷線、廣播失敗、心跳超時都找不到記錄，無法回溯失敗原因。
- root_cause: 所有 `except Exception` 只有 `pass` 或 `break`，沒有寫任何 log。
- wrong_assumption: 長連線的網路例外不重要，不值得記錄。
- checks_next_time:
  - 預期型別（網路中斷、對方主動關閉）→ DEBUG 級別 + 必要 context
  - 非預期例外（協定錯誤、程式邏輯異常）→ WARNING 或 ERROR 級別
  - log 必須帶 tenant_id、connection id 或可追蹤的 context，讓 log 能和 incident 對應
- rule: 任何 `except Exception` block 都必須至少一行 log；允許 `break`/`continue`，但不允許靜默。

### [2026-04-06] 大檔案拆分：循環依賴用 deferred import，向下相容用 re-export
- scope: global
- project: AIHR_LAW
- area: architecture
- symptom: 將 2400+ 行的大檔拆成子模組後，循環 import 導致啟動失敗，舊測試 import 路徑也全部斷掉。
- root_cause:
  - 子模組 B 需要呼叫母模組 A 的函式，但 A 又 include B 的 router → 循環依賴
  - 測試直接 `from app.api.v1.dashboard import list_dashboard_violations`，路徑在拆分後消失
- wrong_assumption: 直接把函式移到子模組後所有 import 都能正常工作。
- checks_next_time:
  - 若子模組需要在「執行期」呼叫母模組的函式（而非 import 期），把 import 放在函式體內（deferred import），避免模組載入時的循環
  - 在母模組頂層加 `from .submodule import symbol  # noqa: F401` re-export，讓舊路徑繼續有效
- rule: 拆大檔時，先列出所有現存的 import 路徑（grep `from app.xxx.原始檔案 import`），確保每一個都在拆分後仍能解析，必要時在母模組加 re-export。

### [2026-04-06] 測試覆蓋率設定必須用 glob，不能用明確檔案清單
- scope: global
- project: AIHR_LAW
- area: testing
- symptom: `vitest coverage` 統計顯示覆蓋率很高，但新增的整個頁面模組完全不在統計範圍內。
- root_cause: `coverage.include` 使用明確檔案清單，新增的 `app/monitor/`、`app/compliance/` 等目錄從未被收入。
- wrong_assumption: 只要列出「重要檔案」就算覆蓋率有效設定。
- checks_next_time: 若 `include` 是一個 array of specific paths，立刻改為 glob；任何新頁面或模組應自動被納入統計
- rule: `coverage.include` 一律使用 glob pattern（如 `"app/**/*.{ts,tsx}"`），禁止維護明確檔案清單；per-file thresholds 另行在 `thresholds` 物件中設定。

### [2026-04-13] pydantic-settings / 任何 env 讀取工具，系統環境變數優先於 `.env`
- scope: global
- project: ESG_HR
- area: environment
- symptom: 後端啟動失敗，報 `SECRET_KEY is set to a known weak default value`；但 `.env` 明明已設定正確強密鑰。
- root_cause: `pydantic-settings`（及大多數 env 讀取工具）的優先順序：OS 層級環境變數 > `.env` 檔。Windows User 層級殘留 `SECRET_KEY=change-me-in-production`，完全蓋過 `.env` 的有效值。
- wrong_assumption: 以為修改 `.env` 就能生效；未意識到系統層級有同名舊值殘留。
- checks_next_time: 啟動失敗且錯誤指向設定驗證時，**先直接查系統環境變數**（PowerShell: `echo $env:VAR_NAME`；bash: `echo $VAR_NAME`），再看 `.env`。
- rule: 任何 env 讀取工具修改 `.env` 後若設定值不如預期，必須先確認系統環境變數層級是否有同名舊值覆蓋；清除或更新系統層級變數才是真正修法，不能只改 `.env`。

### [2026-04-13] HTTP client 請求（前端 fetch / 後端 httpx）必須設定 timeout
- scope: global
- project: ESG_HR
- area: ui
- symptom: 登入頁永遠顯示「登入中...」；即使無依賴的 `/healthz` 也不回應，UI 狀態永久凍結。
- root_cause: 後端 uvicorn worker pool 耗盡/死鎖，所有請求無回應；前端 `fetch()` 沒有 timeout，Promise 永遠懸而未決（pending），UI 無法進入錯誤狀態。
- wrong_assumption: 以為是前端狀態管理 bug 或後端登入邏輯錯誤；沒想到後端整體已卡死且前端沒有 timeout 保護。
- checks_next_time: UI 永久停在 loading，先用 `curl -m 3 http://localhost:PORT/healthz` 確認後端是否存活；若連 healthz 也卡，直接重啟後端程序。
- rule: 所有 HTTP 請求必須設定 timeout：前端用 `AbortController` + `setTimeout` 包裝 `fetch()`（建議 10–15s）；後端用 `httpx.get(..., timeout=N)` 或 `requests.get(..., timeout=N)`；任何無 timeout 的請求在伺服器掛掉時都會造成呼叫端永久凍結。

## 建議紀錄格式

建議固定使用這個 lesson 格式：

```md
### [YYYY-MM-DD] Lesson Title
- scope: global | project
- project: ProjectName
- area: testing | build | mock | database | ui | api | deploy | environment | workflow | encoding | string-corruption | typing | error-handling | architecture | i18n
- symptom:
- root_cause:
- wrong_assumption:
- checks_next_time:
- rule:
```

欄位說明：

- `scope`
  - `global` 代表這條經驗應該進全域母版
  - `project` 代表這條經驗只屬於某個專案
- `project`
  - 寫下經驗來源專案名稱，方便回溯
- `area`
  - 幫助之後篩選同類型 lesson；上列為常用枚舉，可視需要增補（例如 `encoding`、`string-corruption`）。
- `rule`
  - 最重要，這是未來 AI 最應該執行的守則

最小可行版本也可以用：

```md
### [YYYY-MM-DD] Lesson Title
- scope: global | project
- symptom:
- root_cause:
- rule:
```

全域 lesson 範例：

```md
### [2026-04-05] 建置失敗來自輸出目錄被鎖定
- scope: global
- project: ESG_HR
- area: build
- symptom: `vite build` 因 `EPERM` 無法清空輸出目錄而失敗。
- root_cause: 仍有開發伺服器程序占用 `dist` 內檔案。
- wrong_assumption: 誤以為是 TypeScript 或 Vite 設定錯誤。
- checks_next_time: 調整建置設定前，先確認沒有開發伺服器或其他程序占用輸出目錄。
- rule: 建置若出現 `EPERM`、`Permission denied` 或無法清空輸出目錄，應先檢查背景程序與檔案鎖定，再改建置設定。
```

### [2026-04-16] 匯入流程遇到資料庫重複資料時不得直接覆蓋
- scope: global
- project: LOUISA_SHIFT
- area: import / overwrite-confirmation
- symptom: 匯入班表、主檔或其他資料時，資料庫已存在相同編號或唯一鍵資料，但系統沒有明確顯示「資料重複」並讓使用者決定是否覆蓋。
- root_cause: 匯入流程把解析與寫入綁在一起，直接將重複資料視為可自動更新，或在格式誤判時讓使用者誤以為匯入成功卻沒有實際完成確認。
- wrong_assumption: 只要匯入 key 相同就可以直接覆寫，或以為自動覆蓋比要求使用者確認更方便。
- checks_next_time: 任何 import、bulk upsert、sync 或批次寫入流程，都要先比對資料庫中的唯一識別欄位；只要命中重複，就必須回傳 row-level duplicate 狀態、重複摘要與明確覆蓋選項，讓使用者確認後才能覆寫。
- rule: 匯入任何資料時，只要資料庫中已有相同 key、編號或唯一識別資料，一律先顯示「資料重複」並讓使用者決定是否覆蓋；不得在未明確確認前直接覆蓋。
